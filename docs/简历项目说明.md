## 三、核心业务模块

### 1. 二手商品交易

- 首页展示商品列表，支持**关键词搜索**（标题、描述、地点）和**分类筛选**（数码、教材、日用等 6 类）
- 商品详情页：价格、卖家、联系方式、交易地点、在售状态
- 发布商品表单：**服务端校验** + 错误回显
- 内置种子数据，启动即可演示

**涉及文件：**

- `model/Product.java`、`ProductCategory.java`、`ProductStatus.java`
- `repository/ProductRepository.java`
- `templates/index.html`、`product-detail.html`、`publish.html`

### 2. 代课互助

- 代课需求列表与搜索
- 发布代课：课程名、时间、地点、价格、补充说明
- 详情页展示完整信息

**涉及文件：**

- `model/SubstitutePost.java`
- `repository/SubstitutePostRepository.java`
- `templates/substitutes.html`、`substitute-detail.html`、`substitute-publish.html`

### 3. 实时聊天（WebSocket）

- 路径：`/ws/substitutes/{id}`
- 同一代课帖下的多用户**房间广播**
- 消息持久化到 MySQL（`chat_messages` 表）+ 页面初次加载历史消息
- 前端 WebSocket 自动重连提示、昵称 `localStorage` 记忆
- 使用 `ConcurrentHashMap` 管理会话，`ObjectMapper` 序列化 JSON

**涉及文件：**

- `websocket/SubstituteChatWebSocketHandler.java`
- `config/WebSocketConfig.java`
- `model/ChatMessage.java`

**面试可讲点：**

- 连接建立时校验帖子是否存在
- `afterConnectionClosed` 清理空房间，避免内存泄漏
- 同时保留 HTTP POST 发消息作为降级路径

### 4. 微信支付（Native 扫码）

完整支付链路：

```
用户点击「微信支付」
  → 创建 PaymentOrder（订单号 CUZ + 时间戳）
  → 调用微信 Native 下单 API 获取 code_url
  → 跳转支付页，ZXing 生成二维码图片
  → 前端每 3 秒轮询 /payments/{orderNo}/status
  → 微信异步回调 /payments/wechat/notify
  → 验签解析 → 更新订单状态为 PAID / FAILED
```

**技术细节（简历加分项）：**

- 微信支付 API v3：`RSAAutoCertificateConfig` 自动管理平台证书
- 回调通知：`NotificationParser` 验签 + 解析 `Transaction`
- 金额转换：`BigDecimal` → 分（`int`），避免浮点误差
- 配置未就绪时优雅降级（按钮禁用 + 友好提示）

**涉及文件：**

- `payment/WechatNativePayClient.java`
- `config/WechatPayProperties.java`
- `model/PaymentOrder.java`、`PaymentStatus.java`、`PaymentTargetType.java`
- `repository/PaymentOrderRepository.java`
- `templates/payment-detail.html`

### 5. 部署与运维

- **Docker 多阶段构建**：编译阶段 `maven:3.9.9-eclipse-temurin-21`，运行阶段 `eclipse-temurin:21-jre`，镜像体积小
- **GitHub Actions**：`mvn test` → `docker build` → `scp` 传镜像 → 远程 `docker load` + 滚动替换容器
- **Nginx**：`proxy_set_header` 传递真实 IP 和协议头，为后续 HTTPS 做准备

**涉及文件：**

- `Dockerfile`
- `.github/workflows/deploy.yml`
- `deploy/nginx-cuzmall.conf`

## 五、面试高频问题 & 回答思路

### Q1：为什么用 Thymeleaf 而不是前后端分离？

> 项目定位是校园场景 MVP，页面以表单和列表为主，SSR 开发效率高、部署简单（一个 Jar + 静态资源）。支付页和聊天页用少量原生 JS 补足交互。若后续需要移动端或复杂交互，可逐步拆出 REST API + Vue/React 前端。

### Q2：为什么用 JPA 而不是 MyBatis？

> 项目实体关系简单、以 CRUD 和少量 JPQL 搜索为主，JPA 开发效率高。`ProductRepository.search` 用 JPQL 做多字段模糊查询；聊天消息、订单、用户账号统一走 `JpaRepository`。若后续报表或复杂 SQL 增多，可对单表查询改用 `@Query(nativeQuery=true)` 或局部引入 MyBatis。

### Q3：微信支付回调如何保证安全？

> 使用 `wechatpay-java` 的 `NotificationParser`，基于请求头中的 `Wechatpay-Signature`、`Wechatpay-Timestamp`、`Wechatpay-Nonce` 和平台证书验签，解析 body 得到 `Transaction`，再根据 `trade_state` 更新本地订单。未验签通过的请求不会改状态。

### Q4：WebSocket 和 HTTP 发消息为什么两套？

> WebSocket 提供实时体验；HTTP POST 作为连接失败时的备用，也便于不支持 WebSocket 的环境调试。两者最终都调用 `SubstitutePostRepository.addMessage`。

### Q5：部署流程是怎样的？

> push 到 main → Actions 执行 `mvn test` → `docker build` 多阶段镜像 → `docker save` 压缩 → `scp` 到服务器 → 远程 `docker load` → 删除旧容器、启动新容器（`--restart always`）→ `docker image prune` 清理。

### Q6：遇到过什么难点？

可选真实经历（按实际情况选用）：

- 微信商户证书与 APIv3 密钥环境配置繁琐 → 抽象 `WechatPayProperties.isConfigured()` 做启动期检测
- WebSocket 在 Nginx 后需要升级头（若上 HTTPS/WSS 需配置 `proxy_http_version 1.1` 和 `Upgrade`）
- 金额必须用分且整数，用 `BigDecimal` 乘 100 再 `intValueExact` 避免精度问题
